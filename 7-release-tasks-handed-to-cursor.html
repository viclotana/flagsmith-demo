7 Release Tasks I Handed to Cursor Instead of Doing Myself

I wanted to see how much of my release workflow an AI agent could handle autonomously. Here's what happened.

The Experiment

Writing code is only a small part of shipping software. The real work starts once the code exists. Deciding who sees it, when, at what percentage, and how to pull it back if something breaks. That part has always stayed human.

I wanted to test how far that line had actually moved with Cursor.

Before You Start

You'll need two things: a Flagsmith account and the MCP server connected to your AI agent or coding tool.

If you're already using Flagsmith, grab your API key from Organisation Settings. Then connect it to Cursor with one command:

Add the server to ~/.cursor/mcp.json (global) or .cursor/mcp.json (per project):

{
    "mcpServers": {
        "flagsmith": {
            "url": "https://mcp.flagsmith.com",
            "headers": { "Authorization": "Token <your-flagsmith-api-key>" }
        }
    }
}

Restart your agent, run /mcp to confirm the connection, and you're ready. The rest of this article assumes you're set up.

For one feature, a new checkout experience, I decided not to touch the Flagsmith dashboard at all. The goal is to go through a release cycle using Cursor with the same environments and release strategy my team already follows. Just a different interface.

Here's what happened across seven tasks.

1. Create the Release

Every feature starts with a release plan.

Instead of opening Flagsmith and manually creating a feature flag, I described what I wanted. Cursor translated intent into release infrastructure. Flag created, environments confirmed, feature ID returned.

Create a feature flag called checkout_v2 for the new checkout experience. Return the feature ID and tell me which environments it exists in.

What surprised me: I didn't have to think about the UI at all. The mental overhead of navigating to the right screen, filling in the right fields. Completely disappeared.

2. Ship to the Right People

Shipping isn't binary.

One of the biggest advantages of feature flags is controlling who sees a feature before the rest of the world does. Internal users first. Then a small percentage. Then everyone.

Instead of configuring targeting rules manually, I described the intent.

Enable checkout_v2 only for internal users. If an internal_users segment doesn't exist, create one first.

Cursor created the segment, applied the targeting rule, and confirmed the flag state. One prompt. No dashboard.

3. Turn It Into an Experiment

Once a feature is behind a flag, you have a natural opportunity to compare experiences rather than make permanent decisions.

Most teams skip this step because setting up an experiment manually adds friction. So they ship, guess, and move on.

I asked Cursor to set one up instead.

Create an experiment comparing checkout_v1 and checkout_v2. Use checkout_completed as the primary success metric.

The experiment was configured in the same session, without breaking the flow of building the feature. That matters more than it sounds: the reason teams skip experimentation is usually friction, not intent.

4. Understand What's Live

One of the most common release questions isn't about changing something.

It's about understanding what's already running.

Engineers spend more time than they should clicking through environments trying to answer basic questions: what's enabled, who can see it, what changed and when. That work is invisible, low-value, and genuinely annoying.

Show me everything that's currently enabled in production for checkout_v2. Summarise the release state in plain English.

Cursor returned a plain English summary of the current flag state across environments. This was the most useful task of the seven. Not the flashiest, but the one I'd use every single day.

5. Change Production Safely

Production changes shouldn't happen blindly.

Engineering teams have approval workflows for a reason. The value of feature flags isn't just that you can change things. It's that you can change things deliberately, with visibility into what's about to happen before it does.

Rather than asking Cursor to immediately publish changes, I asked it to prepare them first.

Prepare checkout_v2 for production. Show me every proposed change before publishing anything.

Cursor returned a change summary and waited. That one behaviour (prepare, explain, pause) is the difference between an AI agent that helps you move fast and one that gives you a heart attack on a Friday afternoon.

6. Roll Out Gradually

Controlled rollouts are one of the most important risk-reduction tools in software delivery. They're also one of the most skipped, because configuring them manually adds steps to an already long process.

Roll out checkout_v2 to 10% of users in production. Confirm the rollout percentage and flag state after.

Done in the same session. Same conversation. No context switch.

The 10% rollout that used to take three minutes of dashboard navigation happened in one sentence. That's not a small thing when you're in the middle of a deployment and your attention is already split.

7. Roll It Back

Every release strategy should assume something will eventually go wrong.

The final test wasn't deploying. It was recovering. Could Cursor reverse the rollout just as naturally as it initiated it?

Disable checkout_v2 in production. Explain exactly what changed after the rollback.

It could. Flag disabled, change confirmed, state summarised. The rollback that matters most, the one you do at 11pm when something is broken and you're already stressed, took one prompt.

What I Learned

The biggest shift wasn't speed, though everything was faster. It was that I never had to leave the session.

Every release task I described above used to require a context switch. Open a browser, navigate to the right screen, configure the right fields, copy something back into the terminal. Individually, each switch takes thirty seconds. Collectively, across a full release cycle, they add up to something that quietly fragments your attention and slows you down.

Flagsmith MCP

With the Flagsmith MCP server connected to Cursor, the entire release workflow happened inside the same conversation where I was writing code. The tool didn't change. The guardrails didn't change. My release strategy didn't change. The interface did.

That's the actual value. Not that AI can manage feature flags. It's that AI can now participate in software delivery inside the same context where it's already building the software.

Writing code is quickly becoming the easy part. The next generation of developer tools won't just help AI write software. They'll help AI release it safely, inside the guardrails every engineering team already relies on.

That line is moving faster than most people realise.
